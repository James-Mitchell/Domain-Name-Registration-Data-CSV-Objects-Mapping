1.  Introduction

The Registry Data Escrow Specification can be extended to support any type of object that is included in a data escrow. This document defines the data escrow structure of the standard set of objects for a Domain Name Registry which include:

Domain:
Internet domain names that are typically provisioned in a Domain Name Registry using the EPP domain name mapping (Hollenbeck, S., “Extensible Provisioning Protocol (EPP) Domain Name Mapping,” August 2009.) [RFC5731]. The attributes defined in the EPP domain name mapping (Hollenbeck, S., “Extensible Provisioning Protocol (EPP) Domain Name Mapping,” August 2009.) [RFC5731] are fully supported by this document.
Host:
Internet host names that are typically provisioned in a Domain Name Registry using the EPP host mapping (Hollenbeck, S., “Extensible Provisioning Protocol (EPP) Host Mapping,” August 2009.) [RFC5732]. The attributes defined in the EPP host mapping (Hollenbeck, S., “Extensible Provisioning Protocol (EPP) Host Mapping,” August 2009.) [RFC5732] are fully supported by this document.
Contact:
Individual or organization social information provisioned in a Domain Name Registry using the EPP contact mapping (Hollenbeck, S., “Extensible Provisioning Protocol (EPP) Contact Mapping,” August 2009.) [RFC5733]. The attributes defined in the EPP contact mapping (Hollenbeck, S., “Extensible Provisioning Protocol (EPP) Contact Mapping,” August 2009.) [RFC5733] are fully supported by this document.
Registrar:
The organization that sponsors objects like domains, hosts, and contacts in a Domain Name Registry. The Registrar MAY also be associated with contacts, referred to as Registrars contacts.
IDN language rules:
Internationalized Domain Names (IDN) included in the Domain Object Data Escrow include references to the languages rules that define the set of character code points allowed for a specific language.
EPP parameters:
Each Domain Name Registry supports a set of EPP object mappings, EPP extensions, and data collection policy (DCP).

1.1.  Models

This document defines two different models that can be used to deposit data escrow objects that follow the structure defined in the Registry Data Escrow Specification. The models include:

XML
The XML model includes all of the deposit information (meta-data and data) in an XML document. The definition of the XML format is fully defined in the XML schemas. This model is appropriate for small to medium sized registries that can fully validate the data escrow deposit using a standard validation XML parser.
CSV
The CSV model uses XML to define the data escrow format of the data contained in referenced Comma-Seperated Values (CSV) files. This model is appropriate for medium to large sized registries that will be overly expensive to generate and validate a large data escrow deposit. This model MAY also be used where it is more cost effective to generate and consume the CSV data from and to a database.

The data escrow file MAY contain a mix of both models but SHOULD contain a single model.

3.1.  Date and Time

Numerous fields indicate "dates", such as the creation and expiry dates for domains. These fields SHALL contain timestamps indicating the date and time in UTC as specified in [RFC3339] (Klyne, G., Ed. and C. Newman, “Date and Time on the Internet: Timestamps,” July 2002.), with no offset from the zero meridian.


 TOC 

3.2.  Country names

Country identifiers SHALL be represented using two character identifiers as specified in [ISO‑3166‑1] (International Organization for Standardization, “Codes for the representation of names of countries and their subdivisions - Part 1: Country codes,” November 2006.).


 TOC 

3.3.  Telephone numbers

Telephone numbers (both voice and fax) SHALL be formatted based on structures defined in [ITU‑E164] (International Telecommunication Union, “The international public telecommunication numbering plan,” February 2005.). Telephone numbers described in this specification are character strings that MUST begin with a plus sign ("+", ASCII value 0x002B), followed by a country code defined in [ITU‑E164] (International Telecommunication Union, “The international public telecommunication numbering plan,” February 2005.), followed by a dot (".", ASCII value 0x002E), followed by a sequence of digits representing the telephone number.


 TOC 

3.4.  Internationalized and Localized Elements

Some elements MAY be provided in either internationalized form ("int") or provided in localized form ("loc"). Those elements use a "type" or "isLoc" attribute to specify the form used. If an "isLoc" attribute is used, a value of "true" indicates the use of the localized form and a value of "false" indicates the use of the internationalized form. This MAY override the form specified for a parent element. A "type" attribute value of "int" is used to indicate the internationalized form and a value of "loc" is used to indicate the localized form. When the internalized form ("int") is provided, the element value MUST be represented in a subset of UTF-8 that can be represented in the 7-bit US-ASCII character set. When the localized form ("loc") is provided, the element value MAY be represented in unrestricted UTF-8. The <rdeContact:postalInfo> element below is an example of an internationalized element: 

...
    <rdeContact:postalInfo type="int">
      <contact:name>John Doe</contact:name>
      <contact:org>Example Inc.</contact:org>
      <contact:addr>
        <contact:street>123 Example Dr.</contact:street>
        <contact:street>Suite 100</contact:street>
        <contact:city>Dulles</contact:city>
        <contact:sp>VA</contact:sp>
        <contact:pc>20166-6503</contact:pc>
        <contact:cc>US</contact:cc>
      </contact:addr>
    </rdeContact:postalInfo>
...

3.6.1.  <rdeCsv:csv> and <rdeCsv:csvDelete> elements

To support a CSV model with the Registry Data Escrow Specification, a <rdeCsv:csv> element is defined that substitutes for <rde:content> as a child element of a <rde:contents> substitute element like the <rdeDomain:contents> element. A <rdeCsv:csvDelete> element substitutes for <rde:delete> as a child element of the <rde:deletes> element. The <rdeCsv:csv> and <rdeCsv:csvDelete> elements share the following child elements:

<rdeCsv:sep>
Optional definition of the CSV separator characters with the default separator character of ",".
<rdeCsv:fields>
Ordered list of CSV fields used in the CSV files. There is one or more child elements that substitute for the <rdeCsv:field> abstract element. Each element defines the format of the CSV field contained in the CSV files. <rdeCsv:field> elements by convention support the "type" attribute that defines the XML data type of the field elements, an optional "list" attribute that when "true" indicates that the field is a list with the list elements type defined by the "type" attribute, and an optional "sep" attribute for list fields that defines the list element separator characters with a default seperator of ','. List fields MUST be enclosed in double quotes '"'. By convention all fields are optional, where an undefined field can be represented with an empty field value ",," in the CSV file, where ',' is the seperator character.
<rdeCsv:files>
A list of one or more CSV files using the <rdeCsv:file> child element. The <rdeCsv:file> child element defines a reference to the CSV file name and has the following optional attributes:
compression
If the CSV file is compressed, the "compression" attribute defines the compression format like "gzip" or "zip".
encoding
Defines the encoding of the CSV file with the default encoding of "UTF-8".
chksum
Defines the checksum CRC (UNIX cksum) of the CSV file. This attribute is used to validate that the full CSV file exists and has not been tampered with.

The <rdeCsv:csv> and <rdeCsv:csvDelete> elements requires a "name" attribute that defines the purpose of the CSV file with values like "domain", "host", "contact". The supported "name" attribute values are defined for each object type.

The following is an example of the <rdeCsv:csv> element for domain name records.

4.  Protocol Description

The following is a format for Data Escrow deposits for the standard set of objects of a Domain Name Registry. Format elements are reused from the EPP [RFC5730] (Hollenbeck, S., “Extensible Provisioning Protocol (EPP),” August 2009.) and related RFCs.


 TOC 

4.1.  Domain Name Object

The domain name object is based on the EPP domain name mapping in [RFC5731] (Hollenbeck, S., “Extensible Provisioning Protocol (EPP) Domain Name Mapping,” August 2009.). The domain name object supports both the XML Model and the CSV Model, defined in the Models (Models) section. The elements used for both models are defined in the following sections. Both models use the <rdeDomain:contents> child element of the <rde:contents> element to hold the domain name objects for the deposit.


 TOC 

4.1.1.  XML Model

There are two elements used in the data escrow of the domain name objects for the XML model including the <rdeDomain:domain>, under the <rdeDomain:contents> element, and the <rdeDomain:delete> element, under the <rde:deletes> element.


 TOC 

4.1.1.1.  <rdeDomain:domain> element

The <rdeDomain:domain> element is based the EPP Domain Name Mapping (Hollenbeck, S., “Extensible Provisioning Protocol (EPP) Domain Name Mapping,” August 2009.) [RFC5731] for both the domain name object information and domain name transfer information. The DNSSEC information is based on the EPP DNSSSEC Extension (Gould, J. and S. Hollenbeck, “Domain Name System (DNS) Security Extensions Mapping for the Extensible Provisioning Protocol (EPP),” May 2010.) [RFC5910]. The <rdeDomain:domain> element contains the following child elements:

<rdeDomain:name>
Contains the fully qualified name of the domain name object.
<rdeDomain:uName>
An OPTIONAL UTF-8 encoded fully qualified name of the IDN domain name object.
<rdeDomain:roid>
Contains the Repository Object IDentifier assigned to the domain name object when the object was created.
<rdeDomain:idn>
An OPTIONAL boolean flag indicating if the domain name object is an IDN domain name object.
<rdeDomain:language>
An OPTIONAL language tag value for an IDN domain name object. The language tag value MUST be one of the <rdeIDN:language> elements contained in an <rdeIDN:languageRules> element, described in Section 4.5 (IDN Language Rules Object).
<rdeDomain:variant>
An OPTIONAL boolean flag indicating if the domain name object is an IDN variant domain name object. If the <rdeDomain:variant> value is "true" the <rdeDomain:originalName> SHOULD be set with the original IDN domain name object.
<rdeDomain:originalName>
An OPTIONAL fully qualified name of the original IDN domain name object related to the variant domain name object. This value SHOULD be set if <rdeDomain:variant> is "true".
<rdeDomain:reserved>
An OPTIONAL boolean flag indicating if the domain name object is a reserved domain name object. A reserved domain name object is not registered by a client but by the server to ensure that it can't be registered by a client.
<rdeDomain:status>
Zero or more OPTIONAL <rdeDomain:status> elements that contain the current status descriptors associated with the domain name object using the status values defined in [RFC5731] (Hollenbeck, S., “Extensible Provisioning Protocol (EPP) Domain Name Mapping,” August 2009.).
<rdeDomain:rgpStatus>
Zero or more OPTIONAL <rdeDomain:rgpStatus> elements that contain the current Registry Grace Period (RGP) status descriptors associated with the domain name object using the status values defined in [RFC3915] (Hollenbeck, S., “Domain Registry Grace Period Mapping for the Extensible Provisioning Protocol (EPP),” September 2004.).
<rdeDomain:registrant>
An OPTIONAL registrant contact identifier associated with the domain name object.
<rdeDomain:contact>
Zero or more OPTIONAL <rdeDomain:contact> elements that contain the contact identifiers and the contact types as defined in [RFC5731] (Hollenbeck, S., “Extensible Provisioning Protocol (EPP) Domain Name Mapping,” August 2009.) associated with the domain name object.
<rdeDomain:ns>
An OPTIONAL element that contains the fully qualified names of the delegated host objects or host attributes (name servers) associated with the domain name object. Refer to section 1.1 of [RFC5731] (Hollenbeck, S., “Extensible Provisioning Protocol (EPP) Domain Name Mapping,” August 2009.) for a description of the elements used to specify host objects or host attributes.
<rdeDomain:host>
Zero or more OPTIONAL <rdeDomain:host> elements that contain the fully qualified names of the subordinate host objects that exist under this superordinate domain name object.
<rdeDomain:clID>
Contains the identifier of the sponsoring client.
<rdeDomain:crID>
An OPTIONAL element that contains the identifier of the client that created the domain name object.
<rdeDomain:crDate>
An OPTIONAL element that contains the date and time of domain name object creation.
<rdeDomain:exDate>
An OPTIONAL element that contains the date and time identifying the end of the domain object's registration period.
<rdeDomain:upID>
An OPTIONAL element that contains the identifier of the client that last updated the domain name object.
<rdeDomain:upDate>
An OPTIONAL element that contains the date and time of the most recent domain-object modification.
<rdeDomain:trDate>
An OPTIONAL element that contains the date and time of the most recent domain-object transfer.
<rdeDomain:authInfo>
An OPTIONAL element that contains the authorization information associated with the domain name object per [RFC5731] (Hollenbeck, S., “Extensible Provisioning Protocol (EPP) Domain Name Mapping,” August 2009.).
<rdeDomain:dnssec>
An OPTIONAL element containing the DNSSEC information associated with the domain name object per [RFC5910] (Gould, J. and S. Hollenbeck, “Domain Name System (DNS) Security Extensions Mapping for the Extensible Provisioning Protocol (EPP),” May 2010.).
<rdeDomain:transfer>
Zero or more optional <rdeDomain:transfer> elements that contain the information of pending transfer and completed transfer requests.
<rdeDomain:extension>
An OPTIONAL element containing the custom domain name object attribute extensions.

Example of a <rdeDomain:domain> element: 

...
<rdeDomain:contents>
  ...
  <rdeDomain:domain>
     <rdeDomain:name>pinguino.test</rdeDomain:name>
     <rdeDomain:roid>Dpinguino-TEST</rdeDomain:roid>
     <rdeDomain:status s="pendingDelete"/>
     <rdeDomain:rgpStatus s="redemptionPeriod"/>
     <rdeDomain:registrant>jd1234</rdeDomain:registrant>
     <rdeDomain:contact type="admin">sh8013</rdeDomain:contact>
     <rdeDomain:contact type="tech">sh8013</rdeDomain:contact>
     <rdeDomain:ns>
        <domain:hostObj>ns1.example.com</domain:hostObj>
        <domain:hostObj>ns1.example.net</domain:hostObj>
     </rdeDomain:ns>
     <rdeDomain:host>ns1.pinguino.test</rdeDomain:host>
     <rdeDomain:host>ns2.pinguino.test</rdeDomain:host>
     <rdeDomain:clID>clientX</rdeDomain:clID>
     <rdeDomain:crID>clientY</rdeDomain:crID>

     <rdeDomain:upID>clientX</rdeDomain:upID>
     <rdeDomain:upDate>2009-12-03T09:05:00.0Z</rdeDomain:upDate>
     <rdeDomain:authInfo>
        <domain:pw>2fooBAR</domain:pw>
     </rdeDomain:authInfo>
     <rdeDomain:dnssec>
        <secDNS:maxSigLife>604800</secDNS:maxSigLife>

4.1.2.  CSV Model

4.4.  Registrar Object

The registrar object represents the sponsoring client for other objects. The registrar object MAY be related to registrar contacts using Contact Object, as defined in Section 4.3 (Contact Object), references using the <rdeRegistrar:contactObj> element or MAY contain registrar contacts using the <rdeRegistrar:contactAttr> element. The registrar object supports both the XML Model and the CSV Model, defined in Section 1.1 (Models). The elements used for both models are defined in the following sections. Both models use the <rdeRegistrar:contents> child element of the <rde:contents> element to hold the registrar objects for the deposit.

4.5.  IDN Language Rules Object

The IDN domain names, defined in Section 4.1 (Domain Name Object), MAY have references to the applicable IDN language rules using the language as a lookup key with the <rdeDomain:language> element in the XML model or the <rdeDomain:fLanguage> field element in the CSV model. The language rules are logical expressions that reference external URL's for the detailed character code points. Multiple <rdeDomain:language> or <rdeDomain:fLanguage> values can share the same IDN language rule. The IDN Language Rules only supports the XML model, since the definition of the language rules does not work well with the CSV model and the amount of data is relatively small.

There are two elements used in the data escrow of the IDN language rule objects including the <rdeIDN:languageRules>, under the <rdeIDN:contents> element, and the <rdeIDN:delete> element, under the <rde:deletes> element.


 TOC 

4.5.1.  <rdeIDN:languageRules> element

The <rdeIDN:languageRules> element defines the language rules for a set of languages referenced from by the <rdeDomain:language> element or the <rdeDomain:fLanguage> field element using a logical expression. The logical expression defines the language tables, with an external URL reference per language table, that apply to the set of languages. There are two options in selecting the language tables. The first option is when the character code points of the domain name MUST fall into only one of the language tables with the use="one" attribute value. The second option is when the character code points of the domain name MAY fall into any one of the language tables using the use="all" attribute value. Each entry in the list of language tables can be a list with a different "use" attribute value, resulting in a recursive structure. The <rdeIDN:languageRules> element contains the following child elements:

<rdeIDN:language>
One or more <rdeIDN:language> elements representing the language that the rules applies to. The <rdeIDN:language> element values MUST reside in a single <rdeIDN:languageRules> element so that there is no conflict. The <rdeDomain:language> element and the <rdeDomain:fLanguage> field element, defined in Section 4.1 (Domain Name Object) MUST match one and only one <rdeIDN:language> element in the set of <rdeIDN:languageRules>
<rdeIDN:table>
One or more <rdeIDN:table> element that define the set of language tables that the domain name character code points can fall in. The "use" attribute has two different values including "one" and "all". The "one" value specifies that the domain name character code points can only fall in one of the child language table rule elements. The "all" value specifies that the domain name code points can fall in any one of the child language table rule elements. The <rdeIDN:table> element contains the following child elements:
<rdeIDN:url>
Zero or more <rdeIDN:url> elements that reference language table contain the applicable character code points.
<rdeIDN:table>
Zero or more <rdeIDN:table> elements that reference an additional language table rule expression recursively.

Example of a <rdeIDN:contents> element with two language rules across three languages. 

<rde:contents>
  ...
  <rdeIDN:contents>
    <rdeIDN:languageRules>
     <rdeIDN:language>CHI</rdeIDN:language>
     <rdeIDN:table>
      <rdeIDN:url>
      http://www.iana.org/domains/idn-tables/tables/test_zh_1.0.txt
      </rdeIDN:url>
     </rdeIDN:table>
    </rdeIDN:languageRules>
    <rdeIDN:languageRules>
     <rdeIDN:language>LANG1</rdeIDN:language>
     <rdeIDN:language>LANG2</rdeIDN:language>
     <rdeIDN:table use="one">
      <rdeIDN:url>
      http://www.iana.org/domains/idn-tables/tables/test_tab1_1.1.txt
      </rdeIDN:url>
      <rdeIDN:url>
      http://www.iana.org/domains/idn-tables/tables/test_tab2_1.1.txt
      </rdeIDN:url>
      <rdeIDN:table use="all">
        <rdeIDN:url>
        http://www.iana.org/domains/idn-tables/tables/test_tab3_1.1.txt

        </rdeIDN:url>

4.6.  EPP Parameters Object

The OPTIONAL EPP Parameters Object defines the set of object and object extension services supported by the registry, as defined in [RFC5730] (Hollenbeck, S., “Extensible Provisioning Protocol (EPP),” August 2009.). The EPP Parameters Object only supports the XML model and is defined using the <rdeEppParams:contents> element with the <rdeEppParams:eppParams> child element. The EPP Parameters Object SHOULD be included if the registry uses EPP.

The syntax and content of the <rdeEppParams:eppParams> children elements is as explained in section 2.4 of [RFC5730] (Hollenbeck, S., “Extensible Provisioning Protocol (EPP),” August 2009.). The children of the <rdeEppParams:eppParams> are as follows:

  • One or more <version> elements that indicate the EPP versions supported by the registry.
  • One or more <lang> elements that indicate the identifiers of the text response languages supported by the registry's EPP server.
  • One or more <objURI> elements that contain namespace URIs representing the objects that the registry's EPP server is capable of managing.
  • An OPTIONAL <svcExtension> element that contains one or more <extURI> elements that contain namespace URIs representing object extensions supported by the registry's EPP server.
  • A <dcp> element that contains child elements used to describe the server's privacy policy for data collection and management. See section 2.4 of [RFC5730] (Hollenbeck, S., “Extensible Provisioning Protocol (EPP),” August 2009.) for more details.

5.11.  Security Considerations

This specification does not define the security mechanisms to be used in the transmission of the data escrow deposits, since it only specifies the minimum necessary to enable the rebuilding of a Registry from deposits without intervention from the original Registry.

Depending on local policies, some elements or most likely, the whole deposit will be considered confidential. As such the Registry transmitting the data to the Escrow Agent SHOULD take all the necessary precautions like encrypting the data itself and/or the transport channel to avoid inadvertent disclosure of private data.

It is also of the utmost importance the authentication of the parties passing data escrow deposit files. The Escrow Agent SHOULD properly authenticate the identity of the Registry before accepting data escrow deposits. In a similar manner, the Registry SHOULD authenticate the identity of the Escrow Agent before submitting any data.

Additionally, the Registry and the Escrow Agent SHOULD use integrity checking mechanisms to ensure the data transmitted is what the source intended. Validation of the contents by the Escrow Agent is RECOMMENDED to ensure not only the file was transmitted correctly from the Registry, but also the contents are also "meaningful".
